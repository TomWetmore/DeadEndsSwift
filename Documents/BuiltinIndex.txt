These are the built-in functions of the DeadEnds programming language.

The args to built-ins are ParsedExprs. These are tree structures built when the program is parsed. They are static and persistent structures holding the program's expressions. When programs are run the args are evaluated in the current context -- those values are transient ProgramValue object that hold the expressions' run time values.

Args are usually evaluated to ProgramValues as the first step in running a built-in. In a few cases, however, where L-values are needed, an arg is a ParsedExpr .identifier that is not evaluated. The best example is the assignment built-in, "set(identifier, .any)". The identifier must be an .identifier ParsedExpr; it is not evaluated; the identifier is the name of the variable (L-value) that is assigned a new value. The second arg is evaluated and that value is assigned to the identifier.

The DeadEnds language is strongly typed. Every ProgramValue has an explicit type. The types and their associated values are:

.null                -- an explicit null value; no associated value.
.integer(Int)        -- an integer constant; the associated Int is its value.
.double(Double)      -- a double constant; the associated Double is its value.
.boolean(Bool)       -- a boolean constant; the associated Bool is its value.
.string(String)      -- a string constant; the associated String is its value.
.gnode(GedcomNode)   -- a Gedcom node; the assoicated GedcomNode (reference semantics) is its value.
.person(Person)      -- a Person; the associated Person structure (value semantics) is its value.
.family(Family)      -- a Family; the associated Family structure (value semantics) is its value.
.source(Source)      -- future
.list(List)          -- a List of ProgramValues (reference semantics)
.pair(ProgramValue, ProgramValue)      -- a Pair of ProgramValues (now implemented as a .list).
.table(ProgramTable) -- a Dictionary of String to ProgramValue mappings (reference semantics).
.personset(PersonSet<ProgramValue>) -- A set of ...
.traverse(GedcomNode)
.allPersons          -- all Persons in the database
.allFamilies         -- all Families in the database.

The type .any is used in the tables below. There is no .any type -- it signifies arguments that can be of any type.

Unlike ProgramValues variables are not strongly typed. A variable's type is that of the last ProgramValue assigned to it.

Built-in args are parsed as ParsedExpr values. Unless noted, a built-in evaluates each arg to a ProgramValue and checks the resulting runtime type.

The types in the tables below are the required ProgramValue types after evaluation.

Some built-ins require L-values. In those cases the argument must be an identifier ParsedExpr and is not evaluated. Examples: set(), incr(), decr(), and some extract built-ins.

=========================================================
Null Forwarding and Returning Empty Lists.
=========================================================

Many built-ins use null-forwarding. It refers to a built-in returning .null if either a .null is used as an argument, or the information requested is not available. For example the father() function takes a person and returns the person's father. If the person does not have a father the function returns .null. Also, if .null is passed in to father(), then father() returns .null. This allows built-ins to be chained together without extra code that checks for .nulls.

Consider this program:

proc main () {
    set(p, getperson("enter a person"))
    "1 " name(p) nl()
    "2 " name(father(p)) nl()
    "3 " name(father(father(p))) nl()
    "4 " name(father(father(father(p)))) nl()
    "5 " name(father(father(father(father(p))))) nl()
    "6 " name(father(father(father(father(father(p)))))) nl()
}

This asks for a person and then tries to print the person's name and the names of the next five males in the person's paternal line. If I run this on my wife the output is:

1 Luann Frances Grenda
2 Anthony Grenda
3 Jan Grenda
4 Frank Grenda
5
6

because her paternal great great grandfather and beyond is not known. On the fifth call father() is called with Frank Grenda, but it returns .null because Frank's father is not known. On the sixth call father() is called with a .null argument and returns .null. Null forwarding is almost always the right thing to do, and it can greatly simplify the structure of programs.

Some built-ins return lists. For example, children() can be applied to either a person or a family and returns the children of that entity. If the person or family exists but has no children, children() returns an empty list. If the argument itself is .null, children() also returns an empty list. Returning an empty list in this case is a deliberate form of null forwarding for sequence-valued built-ins. The alternative would be to return .null, but empty lists are usually more useful because they work naturally with foreach(), length(), and other list operations.

Summary

• Built-in arguments are ParsedExpr objects. Unless otherwise noted, they
  are evaluated to ProgramValues before the built-in operates on them.

• Types shown in this reference are the required ProgramValue types after
  evaluation.

• .any denotes any ProgramValue. It is documentation shorthand, not an
  actual ProgramValue type.

• Sequence-valued built-ins return empty lists when there are no results.
  Scalar-valued built-ins generally propagate null.

• Lists, Tables, and PersonSets have reference semantics. Mutating a
  structure changes the original object; there is no need to assign it
  back to a variable that references it.

• Some built-ins require an L-value. Their argument must be an identifier
  ParsedExpr and is not evaluated.

Sequence-returning built-ins generally return empty lists, not null, when
there are no results.

.list(.person) means a List whose elements are ProgramValue.person values.
.list(.any) means a List whose elements may be any ProgramValue values.

PersonSet Operations
=====================================================================================

personset    ()                             -> .personset    Create a person set
addtoset     (.personset, .person [, .any]) -> .null         Add a person to a person set
removefromset(.personset, .person)          -> .null         Remove a person from a person set
union        (.personset, .personset)       -> .personset    Nondestructive union
intersect    (.personset, .personset)       -> .personset    Nondestructive intersection
difference   (.personset, .personset)       -> .personset    Nondestructive difference
parentset    (.personset)                   -> .personset    Parent personset of personset
childset     (.peronsset)                   -> .personset    Children set of personset
spouseset    (.personset)                   -> .personset    Spouse set of personset
siblingset   (.personset)                   -> .personset    Sibling set of personset
ancestorset  (.personset)                   -> .personset    Ancestor set of personset
descendentset(.personset)                   -> .personset    Descendant set of personset
namesort     (.personset)                   -> .null         Sort personset by name
keysort      (.personset)                   -> .null         Sort personset by key

Notes:
1. union, intersect, difference are nondestructive -- their argument still exist when the built-in returns (unless the actions of the built-in cause their last reference to dissappear). For example:

    ...
    set(a, personset())
    set(b, personset())
    ... /* Code that adds elements to a and b. */
    set(a, union(a, b))
    ...

    When union completes it returns a new set; sets a and b still exist and a and b still refer to them. But the action of the set causes variable a to now reference the new set. This may cause the original set a to be freed.
2. parentset, childset, spouseset, siblingset, ancestorset, and descendentset do not add the members their arg sets to their result sets. However, members of the arg set may be in the result set if they have the necessary relationship with another member of the args. For example, consider:
    ...                   /* code that creates, s, a set of person set elements. */
    set(t, spouseset(s))  /* t generally does not contain elements from s unless s has persons who are their own spouses). */
    set(t, spouseset(t))  /* t must contain all elements from s and likely many more (t contains all spouses of spouses of s, which includes all of s. */
3. Elements of person sets have an optional .any "value" property. It can be any program value. Algorithms can take advantage of this extra property in any way that works. The value can only be assigned when a person is explicilty added to a set. The set operations do not have a way to compute new values for the elements that get added to the result sets (the DeadEnds programming language does not have any real features for functional programming)
4. It may be useful to add a valuesort function.

===============================================================================
Meta Operations. These are for debugging during program development.
===============================================================================

showframe() -> .string            Shows the current run time stack frame
showstack() -> .stribng           Shows the full run time stack.
valueof(.any) -> .string          Evaluates the argument and shows its type and value.

Notes:
1. When procs and funcs are called, their args are evaluated in the caller's frame. A new frame is made for the called module, and the arg values are assigned to the params there. The new frame is added to the stack, and the called module runs. Identifiers defined in the called module are stored in its frame. When a module returns its frame dissappears and the calling module continues.
2. Meta operations let you inspect the local frame (showframe), the entire run time stack (all frames plus the global table) or the value and type of any expression.

============================================================================
Lists of Lists, Tables of Tables
============================================================================

Lists and Tables are reference ProgramValues. This simplifies internal operations, and it arbitrary structuring. For example, a .list holds a list of ProgramValues with no restrictions on what the ProgramValues can be; for example, they can be .list or .table objects of their own.

There is no requirement that the elements in a .list or the values in a .table have the same type. Each element or value must have a type, but that happens automatically.

Of course, if you go crazy with "intersting" structures, and structure with mixed types, you will deserve the problems they cause.
