These are the built-in functions of the DeadEnds programming language. I wrote this for both my own use as designer and developer so it contains more internal information than a users guide would normally have.

The arguments to built-ins are ParsedExprs. These are tree structures built when the program is parsed. They are static and persistent representations of the program's expressions. When programs are run the ParsedExprs are evaluated in the context of the database and run time stack. The result of evaluating a ParsedExpr is a transient ProgramValue object that holds the expression's current run time value.

In most cases the ParsedExprs are evaluated to ProgramValues as the first step in running a built-in. In a few cases, however, where L-values are needed, a ParsedExpr must be an .identifier, and it is not evaluated. The best example is the assignment built-in, "set(identifier, .any)". Here identifier must be an .identifier ParsedExpr, and it is not evaluated. The second argument can be any ParsedExpr; the set built-in evaluates it to a ProgramValue, and that value is then assigned to the identifier in a symbol table.

The DeadEnds language is strongly typed in the sense that every ProgramValue has an explicit type. The types and their associated values include:

.null                -- an explicit null value; no associated type.
.ident(String)       -- an identifier (variable); the associated String is its name.
.integer(Int)        -- an integer constant; the associated Int is its value.
.double(Double)      -- a double constant; the associated Double is its value.
.boolean(Bool)       -- a boolean constant; the associated Bool is its value.
.string(String)      -- a string constant; the associated String is its value.
.gnode(GedcomNode)   -- a Gedcom node; the assoicated GedcomNode (reference semantics) is its value.
.person(Person)      -- a Person; the associated Person structure (value semantics) is its value.
.family(Family)      -- a Family; the associated Family structure (value semantics) is its value.
.source(Source)      -- future
.list(List)          -- a List of ProgramValues (reference semantics)
.table(ProgramTable) -- a Dictionary of String to ProgramValue mappings (reference semantics).
.personset(PersonSet<ProgramValue>) -- A set of ...
.traverse(GedcomNode)
.allPersons          -- all Persons in the database
.allFamilies         -- all Families in the database.

The type .any is used in the tables below. There is no .any type; it signifies arguments that can be of any type.

Unlike ProgramValues variables are not strongly typed. A variable's type is that of the last ProgramValue assigned to it.

Built-in arguments are parsed as ParsedExpr values. Unless otherwise noted,
the built-in first evaluates each argument to a ProgramValue, then checks
the resulting runtime type.

The types shown in the tables below are the required ProgramValue types after evaluation.

Some built-ins require L-values. In those cases the argument must be an
identifier ParsedExpr and is not evaluated. Examples: set(), incr(), decr(),
and some extract-style built-ins.

=========================================================
Null Forwarding
=========================================================

Many built-ins use null-forwarding. For example the father() function is intended to take a person as an argument and return that person's father as a result. If the person does not have a father the function returns .null instead. And also, if .null is passed in to father(), then it also returns .null. This allows built-ins to be very conviently chained together without needing any extra code that checks for .null returns.

Consider this program:

proc main () {
    set(p, getperson("enter a person"))
    "1 " name(p) nl()
    "2 " name(father(p)) nl()
    "3 " name(father(father(p))) nl()
    "4 " name(father(father(father(p)))) nl()
    "5 " name(father(father(father(father(p))))) nl()
}

This asks for a person and then tries to print the person's name and the names of the next four males in the person's paternal line. If I run this on my wife the output is:

1 Luann Frances Grenda
2 Anthony Grenda
3 Jan Grenda
4 Frank Grenda
5

because her paternal great great grandfather is not in the database. Null forwarding is almost always the right thing to do and it greatly simplifies the structure of DeadEnds programs.

Summary

• Built-in arguments are ParsedExpr objects. Unless otherwise noted, they
  are evaluated to ProgramValues before the built-in operates on them.

• Types shown in this reference are the required ProgramValue types after
  evaluation.

• .any denotes any ProgramValue. It is documentation shorthand, not an
  actual ProgramValue type.

• Sequence-valued built-ins return empty lists when there are no results.
  Scalar-valued built-ins generally propagate null.

• Lists, Tables, and PersonSets have reference semantics. Mutating a
  structure changes the original object; there is no need to assign it
  back to a variable that references it.

• Some built-ins require an L-value. Their argument must be an identifier
  ParsedExpr and is not evaluated.

Sequence-returning built-ins generally return empty lists, not null, when
there are no results.

.list(.person) means a List whose elements are ProgramValue.person values.
.list(.any) means a List whose elements may be any ProgramValue values.

Notes.
1. eq and ne use ProgramValue's == operator. So it is only as good as that operator.

non-simple comparisons. This may be an issue for the future.

Logical Operators
=========================================================================

and(.any [, .any]...) -> .bool     Return the result of and'ing up to 32 boolean values.

Notes:

2. not also can have .any argument because it uses the same toBool function as

===============================================================================
Meta Operations. These were added to help debugging.
===============================================================================

showframe() -> .string            Shows the current run time stack frame
showstack() -> .stribng           Shows the full run time stack.
valueof(.any) -> .string          Evaluates the argument and shows its type and value.

Notes:
1. The DeadEnds programming language is conventional. When procedures and functions are called arguments (ParsedExprs) are evaluated in the calling module's stack frame. Then a new frame for the called module is created and the argument values (now ProgramValues) are assigned to their corresponding parameter identifier in the new frame. The new frame is added to the run time stack and the called module is run. Variables defined in the called module are stored in the local symbol table. When a module returns its frame (which holds the symbol table) is removed and the calling module continues where it left off.
2. The meta operations let you inspect the local frame (showframe), the entire run time stack (showstack) or the value and type of any ProgramValue.

============================================================================
Lists of Lists, Tables of Tables
============================================================================

Lists and Tables are reference ProgramValues. This simplifies internal operations, and it allows interesting structuring. For example, a .list holds a list of ProgramValues, and there are no restrictions on what those ProgramValues can be; for example, they can be .list or .table objects of their own.

Of course, if you go crazy with "intersting" structuring, you will deserve all the problems you cause yourself.

There is no requirement that the elements in a .list or the values in a .table have the same type. The only requirement is that each element or value have a type, but you get that for free.
