That separate runner app sounds like exactly the right test vehicle: small surface area, fast startup, and no editing complications. I’ll frame this as a practical porting checklist rather than an abstract iOS tutorial. Yes. For your runner app, an iPad port is very plausible.

1. Private file system.
You mostly get this automatically. Every iPad app has a sandbox with its own Documents, Library, and tmp directories. You do not “create a file system”; you use URLs such as:
FileManager.default.urls(for: .documentDirectory, in: .userDomainMask)[0]
For a first version, I would use the app’s Documents folder for GEDCOM and program files.

2. Getting files onto the iPad.
Likely easiest options:
Use SwiftUI’s fileImporter to pick GEDCOM/program files from Files, iCloud Drive, Dropbox, etc.

Then either:

// read in place
let text = try String(contentsOf: url)

or copy the selected file into your app’s Documents folder.

You can also enable iTunes/Finder file sharing so your app’s Documents folder appears in Finder when the iPad is connected. That is controlled by Info.plist keys, especially UIFileSharingEnabled and often LSSupportsOpeningDocumentsInPlace.

3. Sharing the app without App Store

Yes, but with limits.

For yourself: plug in the iPad, select it as the run destination in Xcode, build/run.

For a few testers: use TestFlight or Ad Hoc distribution. Ad Hoc requires registering device UDIDs. TestFlight is usually easier for real testers, though it goes through App Store Connect. Apple documents both beta/release distribution and registered-device distribution.

For broad public use without App Store: not really, unless you are in special enterprise/internal distribution situations.

4. Xcode project setup

You have two reasonable paths:

Add an iPad/iOS app target to the same project, link it with DeadEndsLib, and share most SwiftUI files.

Or make a multiplatform app target if the Mac/iPad versions are nearly identical. Apple supports both adding new targets and configuring multiplatform app targets.

For your case, I would start with a separate iOS/iPadOS target. Keep DeadEndsLib shared. Keep ProgramPage mostly shared if possible, but make platform-specific file-loading wrappers:

#if os(macOS)
// NSOpenPanel
#elseif os(iOS)
// fileImporter
#endif

My iPad is connected to my computer now. What can I do to see the file system on the iPad?

Apple doesn’t really let you browse the entire iPad file system the way you would on a Mac. But there are several useful things you can see.
1. Finder
With the iPad connected:
1. Open Finder.
2. Look in the sidebar under Locations.
3. Click your iPad.
You’ll see information about the device, storage usage, backups, files shared by apps, etc. If an app supports file sharing, you’ll find it under the Files tab when the iPad is selected.

2. Files App on the iPad
On the iPad itself:
1. Open Files.
2. Tap Browse.

You should see locations such as:
* iCloud Drive
* On My iPad
* Downloads
* Third-party providers (Dropbox, OneDrive, etc.)

“On My iPad” is the closest thing to a visible local file system.

3. Xcode Device Browser. If Xcode is running:

1. Choose Window → Devices and Simulators.
2. Select your iPad.

From there you can:
* Install apps
* View logs
* Download app containers (for apps you build)
This becomes particularly useful once you have your runner app installed.

4. What you’ll see for your own app. After you build an iPad version of the runner and install it, the app gets a sandbox like:

Documents/
Library/
tmp/

You normally don’t see these directly. However, if you enable:

UIFileSharingEnabled = YES
LSSupportsOpeningDocumentsInPlace = YES

then Finder and the Files app can expose your app’s Documents folder. At that point you can drag GEDCOM files directly into your app’s storage from the Mac.

What is "fileImporter"?
=======================================================================
fileImporter is a SwiftUI modifier that displays Apple’s standard document picker.

Think of it as the iPad/iPhone equivalent of NSOpenPanel on the Mac.

On macOS you might do something like:

let panel = NSOpenPanel()
panel.runModal()

On iPadOS, you generally don’t do that. Instead, SwiftUI presents the Files app interface and lets the user choose a document.

A minimal example looks like:

@State private var showingImporter = false

Button("Load GEDCOM") {
    showingImporter = true
}
.fileImporter(
    isPresented: $showingImporter,
    allowedContentTypes: [.data]
) { result in
    switch result {
    case .success(let url):
        print(url)

    case .failure(let error):
        print(error)
    }
}

When the button is pressed, the user sees a picker similar to:

Browse
    iCloud Drive
    On My iPad
    Working Copy
    Dropbox
    ...

and can navigate to a file.

For your runner app, you’d probably have something more like:

.fileImporter(
    isPresented: $showingGedcomImporter,
    allowedContentTypes: [.gedcom]
)

except Apple doesn’t define a GEDCOM type, so initially you might simply allow:

allowedContentTypes: [.text]

and then inspect the filename extension yourself.

The result is a URL:

case .success(let url):

You can read the file directly:

let text = try String(contentsOf: url)

or copy it into your app’s Documents folder.

One nice aspect is that the exact same fileImporter also works on macOS. So if your current runner uses NSOpenPanel, you could eventually replace it with a cross-platform solution:

#if os(macOS)
...
#else
...
#endif

or even use fileImporter on both platforms.

For your first iPad port, I would expect fileImporter to be the only new file-management API you absolutely need to learn. Everything else can come later.
