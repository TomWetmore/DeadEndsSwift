Yes. For your runner app, an iPad port is very plausible.

1. Private file system

You mostly get this automatically. Every iPad app has a sandbox with its own Documents, Library, and tmp directories. You do not “create a file system”; you use URLs such as:

FileManager.default.urls(for: .documentDirectory, in: .userDomainMask)[0]

For a first version, I would use the app’s Documents folder for GEDCOM and program files.

2. Getting files onto the iPad

Likely easiest options:

Use SwiftUI’s fileImporter to pick GEDCOM/program files from Files, iCloud Drive, Dropbox, etc.

Then either:

// read in place
let text = try String(contentsOf: url)

or copy the selected file into your app’s Documents folder.

You can also enable iTunes/Finder file sharing so your app’s Documents folder appears in Finder when the iPad is connected. That is controlled by Info.plist keys, especially UIFileSharingEnabled and often LSSupportsOpeningDocumentsInPlace.

3. Sharing the app without App Store

Yes, but with limits.

For yourself: plug in the iPad, select it as the run destination in Xcode, build/run.

For a few testers: use TestFlight or Ad Hoc distribution. Ad Hoc requires registering device UDIDs. TestFlight is usually easier for real testers, though it goes through App Store Connect. Apple documents both beta/release distribution and registered-device distribution.

For broad public use without App Store: not really, unless you are in special enterprise/internal distribution situations.

4. Xcode project setup

You have two reasonable paths:

Add an iPad/iOS app target to the same project, link it with DeadEndsLib, and share most SwiftUI files.

Or make a multiplatform app target if the Mac/iPad versions are nearly identical. Apple supports both adding new targets and configuring multiplatform app targets.

For your case, I would start with a separate iOS/iPadOS target. Keep DeadEndsLib shared. Keep ProgramPage mostly shared if possible, but make platform-specific file-loading wrappers:

#if os(macOS)
// NSOpenPanel
#elseif os(iOS)
// fileImporter
#endif

My practical recommendation: port the runner first, not the full app. The runner has exactly the right shape for iPad: load GEDCOM, load program, run, view output.
