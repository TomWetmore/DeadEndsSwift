How different would the GedcomTreeView be if the ability to edit the tree were added.

The GedcomTreeView now behaves like a read-only inspector, with expand/collapse and some editing coming indirectly from the form. If you wanted to make it a true editor of the tree itself, here‚Äôs what changes:

1. Interactivity
‚Ä¢ Now: Nodes are drawn, maybe expandable, but not directly mutable.
‚Ä¢ Editor mode: Each node needs affordances:
‚Ä¢ Add child (+ button next to a node).
‚Ä¢ Add sibling (often a ‚Äú‚Ä¶‚Äù menu or secondary +).
‚Ä¢ Remove node (trash icon).
‚Ä¢ Edit tag or value inline (text fields instead of labels).
This turns the tree view into a hybrid of outline view + form.

2. Data binding
‚Ä¢ Now: GedcomTreeView just renders a GedcomNode tree.
‚Ä¢ Editor mode:
‚Ä¢ You need Binding<GedcomNode> or @ObservedObject so the tree view can mutate nodes directly.
‚Ä¢ Or (recommended) the view doesn‚Äôt mutate nodes itself, but calls back into the ViewModel (PersonEditorViewModel) with ‚Äúadd/remove/edit‚Äù requests. That way you centralize mutation logic (so undo/redo and validation can hook in).

3. Identity & stability
‚Ä¢ With editing, stable UUIDs per node become non-negotiable. Otherwise SwiftUI won‚Äôt know what changed versus what was rebuilt. You‚Äôd see whole subtrees collapsing/re-expanding constantly.
‚Ä¢ This is where your earlier UUID decision pays off. Each node needs an id: UUID.

4. Expansion logic
‚Ä¢ Expansion state gets trickier:
‚Ä¢ Removing a node should also clean up its expansion entry.
‚Ä¢ Adding a node should add a fresh UUID but not disturb neighbors.
‚Ä¢ Undo/redo will restore UUIDs so expansion can be preserved.

5. Undo/redo
‚Ä¢ If you let the tree view directly edit the model, you need to hook every mutation into the undo manager.
‚Ä¢ Safer pattern: tree view says ‚Äúuser tapped addChild(tag: ‚ÄúOCCU‚Äù) on node X‚Äù ‚Üí view model executes the operation (logging an undo delta). That way the undo/redo system doesn‚Äôt live in the view.

How different would it look?
‚Ä¢ UI additions: buttons/menus for add/remove/edit inline.
‚Ä¢ Data changes: introduce bindings or callbacks for edits.
‚Ä¢ Identity: UUIDs in every node so SwiftUI can reconcile changes.
‚Ä¢ Undo/redo support: centralize mutations in the VM.

Structurally, the GedcomTreeView isn‚Äôt that different: it‚Äôs still a recursive ForEach over children. But you‚Äôd swap out Text(tag) for TextField(tag) and attach add/remove buttons.

So conceptually: the skeleton stays the same, but every row gets action affordances and stable IDs.

Example SwiftUI skeleton

struct GedcomTreeView: View {
    @ObservedObject var viewModel: PersonEditorViewModel
    var node: GedcomNode

    var body: some View {
        DisclosureGroup(isExpanded: Binding(
            get: { viewModel.expanded.contains(node.id) },
            set: { isExpanded in
                if isExpanded {
                    viewModel.expanded.insert(node.id)
                } else {
                    viewModel.expanded.remove(node.id)
                }
            }
        )) {
            // Recursive children
            ForEach(node.children, id: \.id) { child in
                GedcomTreeView(viewModel: viewModel, node: child)
            }
        } label: {
            HStack {
                // Editable tag
                TextField("TAG", text: Binding(
                    get: { node.tag },
                    set: { newVal in
                        viewModel.updateTag(for: node, newTag: newVal)
                    }
                ))
                .frame(width: 80)

                // Editable value
                TextField("Value", text: Binding(
                    get: { node.val ?? "" },
                    set: { newVal in
                        viewModel.updateValue(for: node, newValue: newVal)
                    }
                ))

                Spacer()

                // Add/remove buttons
                Button(action: { viewModel.addChild(to: node) }) {
                    Image(systemName: "plus.circle")
                }
                Button(action: { viewModel.remove(node) }) {
                    Image(systemName: "trash")
                }
            }
        }
    }
}

ViewModel hooks

The view itself never mutates the tree directly; it calls the VM:

        node.tag = newTag

Is DisclosureGroup a real thing?

DisclosureGroup is a real SwiftUI type, introduced back in iOS 14 / macOS 11. Think of it as SwiftUI‚Äôs built-in ‚Äúexpand/collapse section,‚Äù similar to a row in Finder that has a triangle you can click to show children.

Declaration

struct DisclosureGroup<Label, Content> where Label : View, Content : View

It‚Äôs a container with:
‚Ä¢ a label (the always-visible header row),
‚Ä¢ and content (shown or hidden when expanded).

Simple Example

DisclosureGroup("More Details") {
    Text("Line 1")
    Text("Line 2")
}

This gives you a label ‚ÄúMore Details‚Äù with a disclosure triangle, and expanding reveals two text lines.

Controlling expansion

You can bind expansion to your own state:

@State private var expanded = false

DisclosureGroup("Options", isExpanded: $expanded) {
    Text("Option A")
    Text("Option B")
}

That way you can programmatically expand/collapse, or track which groups are open.

Why it‚Äôs handy for you
‚Ä¢ It‚Äôs recursive by design ‚Üí perfect for displaying hierarchical trees like GEDCOM nodes.
‚Ä¢ You can bind expansion state to your expanded: Set<UUID> model instead of a single Bool, letting you persist which nodes are open.
‚Ä¢ It looks native on macOS/iOS (triangle expanders).

An affordance is a visible clue in an interface that indicates what actions are possible.

Examples:
‚Ä¢ A ‚Äú+‚Äù button next to a node ‚Üí affords adding a child.
‚Ä¢ A trash can icon ‚Üí affords deleting that node.
‚Ä¢ A TextField instead of a Text ‚Üí affords editing the tag or value.
‚Ä¢ A chevron/triangle on the left ‚Üí affords expanding/collapsing children.
Without the clues, the user might not realize they can add, delete, or edit.

struct GedcomTreeView: View {

    var node: GedcomNode

        if node.children.isEmpty {

                        } else {
                            viewModel.expanded.remove(node.id)
                        }
                    }

                content: {
                    // Recursive child nodes
                    ForEach(node.children, id: \.id) { child in
                        GedcomTreeView(viewModel: viewModel, node: child)
                            .padding(.leading, 20)
                    }
                },
                label: {
                    GedcomRow(viewModel: viewModel, node: node)
                }
            )
        }
    }

extension PersonEditorViewModel {
