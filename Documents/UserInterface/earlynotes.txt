Here is the hard part: SwiftUI selection is asynchronous. The old LifeLines version could block in curses. In SwiftUI, you probably do not want the interpreter freezing while waiting for a panel.

The simple first implementation: make getperson("wetmore") search and return the first/best match, or throw if ambiguous. This tests the builtin plumbing but does not solve UI.

The real implementation: make the interpreter capable of suspending at an interactive builtin, show a selection panel, then resume when the user chooses. That is a bigger project.

    You have drawn the problem exactly right. The hard part is the “magic interface,” and in SwiftUI that means suspend the program, show UI, then resume.

I would organize it with three pieces.

1. Interpreter returns a “waiting for user” result

Instead of chooseperson directly returning a ProgramValue, it should create a request:

struct PersonChoiceRequest {
    let candidates: [Person]
    let line: Int
}

Then the interpreter stops with something like:

enum InterpResult {
    case okay
    case returning(ProgramValue?)
    case breaking
    case continuing
    case error
    case waitingForPersonChoice(PersonChoiceRequest)
}

 So chooseperson(...) says:

2. ProgramModel owns the pending request

ProgramModel gets:

var pendingPersonChoice: PersonChoiceRequest?
var isChoosingPerson: Bool { pendingPersonChoice != nil }

When the interpreter returns .waitingForPersonChoice(request), the model stores it:

pendingPersonChoice = request

Then ProgramPage shows a .sheet.

3. The picker sends the chosen person back

The picker view receives candidates:

        model.resumeProgram(with: .person(person))
    },
    onCancel: {
        model.resumeProgram(with: .null)

Then the model resumes the interpreter.

The catch: your interpreter is currently not resumable. Once it returns from interpretProgram(), the Swift call stack is gone. So you have two choices.

Practical first version

Do not suspend/resume yet. Instead, make chooseperson temporarily return the candidate list, as you already did. Then use this to build the picker UI separately.

Real version

You need to make the interpreter resumable. That means one of:

1. Implement an explicit interpreter continuation state.
2. Run the program in an async context and make chooseperson await the picker result.
3. Throw a special “need input” signal and later rerun/replay with stored answer.

For SwiftUI, the cleanest long-term approach is probably async:

protocol ProgramInteraction {
    func choosePerson(from candidates: [Person]) async -> Person?
}

Then:

func bltinChoosePerson(_ args: [ParsedExpr]) async throws -> ProgramValue

But this means the interpreter becomes async all the way up:

try await program.interpretProgram()

That is a big change.

So my advice is staged:

First build:

or rename it:

findpersons(pattern) -> list

Then build the SwiftUI picker using the same [Person].

Then, when ready, convert the interpreter to async and make the real:

chooseperson(pattern) -> person/null

On the UI side, ProgramModel implements the protocol by showing a sheet and waiting:

@MainActor
final class ProgramModel: ProgramInteraction {
    var pendingPersonChoices: [Person] = []
    var isShowingPersonPicker = false

    private var personContinuation: CheckedContinuation<Person?, Never>?

    func choosePerson(from candidates: [Person], prompt: String) async -> Person? {
        pendingPersonChoices = candidates
        isShowingPersonPicker = true

        return await withCheckedContinuation { continuation in
            personContinuation = continuation
        }
    }

    func choose(_ person: Person?) {
        isShowingPersonPicker = false
        personContinuation?.resume(returning: person)
        personContinuation = nil
    }
}

The sheet calls:

model.choose(nil)

So the full flow is:

DeadEnds program calls chooseperson()
→ builtin makes candidate list
→ await interaction.choosePerson(...)
→ ProgramModel shows picker sheet
→ interpreter is paused, UI remains alive
→ user chooses/cancels
→ continuation resumes
→ builtin returns .person or .null
→ interpreter continues

That is the mental model. await is the suspension point; the continuation is the little bridge that lets SwiftUI hand the answer back.
